# 学習記録ルール

すべての説明・解説・コード例は、**ステップごとのMDファイル**に記録して受講生が後で振り返れるようにすること。
**チャットには簡潔な報告のみを表示し、詳細はステップごとのMDファイルに保存する。**

---

## 基本方針

### チャットに表示するもの（簡潔に）
- ✅ ステップが完了したこと
- ✅ 何をしたか（箇条書きで簡潔に）
- ✅ 次に何をするか
- ✅ **詳細を保存したファイルパス**

### ステップMDファイルに保存するもの（詳細に）

各ステップのMDファイルには、**他のルール（teaching.md、hint.md、reference-docs.md）に基づいた充実した内容**を保存する：

#### teaching.mdに基づく内容
- **考え方の説明**: 何をしようとしているか、どういうアプローチがあるか
- **概念の説明**: 専門用語を簡単な言葉で補足
- **例え話**: 分かりやすい比喩を使った説明
- **なぜそうなるのか**: 受講生が理解できるように背景を説明

#### hint.mdに基づく内容
- **段階的なヒント**: 方向性 → 具体的 → あと一歩
- **エラー対応**: エラーメッセージの意味 → 原因 → ヒント → 解決方法
- **調べ方のヒント**: 「〇〇を調べてみてください」

#### reference-docs.mdに基づく内容
- **公式ドキュメントのURL**: 重要な構文・メソッド・パターンごとに
- **公式推奨の書き方**: ベストプラクティス
- **URLの信頼性**: WebSearchで確認した正確なURL

#### その他の詳細内容
- コードの詳細な解説（行ごとの説明）
- コード例とその解説
- よくあるエラーと対処法
- Tips・ベストプラクティス
- 実行コマンドとその説明

---

## アウトプットフォルダ構造

課題ごとに専用フォルダを作成し、**フェーズ → ステップ** の階層で管理する。

```
learning-records/
└── <課題名>/
    ├── progress.md          # ★課題開始時に最初に作成。フェーズ・ステップの全体一覧
    ├── phase1/              # フェーズ1のフォルダ
    │   ├── step_1/          # ステップ1のフォルダ
    │   │   ├── lesson.md    # 詳細説明（学んだことを含む）
    │   │   └── code/        # コード例
    │   ├── step_2/
    │   │   ├── lesson.md
    │   │   └── code/
    │   └── ...
    ├── phase2/              # フェーズ2のフォルダ
    │   ├── step_1/
    │   │   ├── lesson.md
    │   │   └── code/
    │   └── ...
    └── ...
```

### フォルダ名の規則
- 課題フォルダ: 英数字とハイフン、アンダースコアのみ使用
  - 例: `janken-game`, `todo-app`, `portfolio`
- フェーズフォルダ: `phase1`, `phase2` の形式
- ステップフォルダ: `step_N` の形式（Nは数字）

---

## progress.md のフォーマット

**課題を始める一番最初に** `learning-records/<課題名>/progress.md` を作成する。
これが存在しない状態でステップを進めてはいけない。

### MVPファースト原則

**`progress.md` を作成するとき、MVPまでの全フェーズを最初から定義する。**

- **MVP（Minimum Viable Product）**: 最小限の機能で「動くもの」が完成する状態
- 後からフェーズを追加するのではなく、最初の段階で全体像を決めて書いておく
- 全フェーズが最初から見えることで、今どこにいるかを常に把握できる
- 途中でゴールが変わった場合は `progress.md` を更新してよい

**推奨するフェーズ設計の考え方**:
1. **まず「動くもの（MVP）」を定義する** — 最低限何ができれば完成といえるか
2. **MVPをフェーズに分解する** — 小さく区切って1フェーズずつ完成させる
3. **MVPより先の拡張は後続フェーズに** — 最初から盛り込もうとしない

---

**目的**: 課題全体のフェーズ・ステップを一目で把握できるチェックリスト

```markdown
# 📋 <課題名> - 進捗タスクリスト

**ゴール**: <最終的にどうなればOKか（MVP）>
**開始日**: YYYY-MM-DD

---

## Phase 1: <フェーズ名>
**ゴール**: <このフェーズで達成すること>
**進捗**: 0 / X ステップ完了

- [ ] **ステップ1**: <ステップ名> ← 今ここ
- [ ] **ステップ2**: <ステップ名>
- [ ] **ステップ3**: <ステップ名>
...

---

## Phase 2: <フェーズ名>
**ゴール**: <このフェーズで達成すること>
**進捗**: 0 / X ステップ完了

- [ ] **ステップ1**: <ステップ名>
- [ ] **ステップ2**: <ステップ名>
...

---

## Phase N: <フェーズ名>（MVPより先の拡張・任意）
**ゴール**: <このフェーズで達成すること>
**進捗**: 0 / X ステップ完了

- [ ] **ステップ1**: <ステップ名>
...

---

## 🎉 全フェーズ完了！
**完了日**: （完了時に記入）

詳細な学習内容は各ステップの `phaseN/step_N/lesson.md` を参照してください。
```

### ステップ完了時の更新

1. チェックボックスにチェック: `- [ ]` → `- [x]`
2. フェーズの進捗カウントを更新: `0 / X` → `1 / X`
3. "← 今ここ" を次のステップに移動
4. フェーズが完了したら次のフェーズの最初のステップに "← 今ここ" を移動

**重要**: 学んだことの詳細は `phaseN/step_N/lesson.md` に記録する（progress.md には書かない）

## チャットでの報告フォーマット

ステップ完了時、チャットには以下のように簡潔に報告する：

```markdown
## ✅ PhaseN / ステップN 完了

**やったこと**:
- 〇〇を作成
- △△を実装

**ブラウザで確認**:
- http://127.0.0.1:5500/...

**次のステップ**:
□□を実装します

📝 **詳細**: `learning-records/<課題名>/phaseN/step_N/lesson.md` を確認してください
```

- ブラウザで確認できるURLは**必ず記載**する
- 開発サーバーの起動コマンドもあわせて記載する

## 記録するタイミング

### 課題開始時（★最優先）
**他の何より先に `progress.md` を作成する。これが最初の作業。**

1. `learning-records/<課題名>/` フォルダを作成
2. **`progress.md` を作成** — 以下の順序で考えてから書く：
   - **① MVPを決める**: 最低限どんな状態になれば「完成」か定義する
   - **② MVPまでのフェーズを全部書く**: Phase 1・2…を最初から洗い出す
   - **③ 各フェーズのステップを5-8個に分ける**: フェーズごとに細かい手順を定義
   - **④ MVPより先の拡張フェーズがあれば末尾に追加**（任意）
   - **最初のステップは必ず「環境構築・ブラウザ確認方法のセットアップ」とする**
   - 1ステップずつ進め、現在のステップだけに集中する
3. `progress.md` の全フェーズ・全ステップを確認してから実装を開始する

#### 環境構築ステップで行うこと
- 使用する技術スタックに合った開発サーバーを用意する
- ブラウザで実際にページが表示できることを確認する
- 以降のすべてのステップで、完了後にブラウザで表示確認を行う

#### 技術スタック別：推奨する環境構築

**HTML / CSS / JavaScript（Node.js不要）**
→ VS Code の **Live Server 拡張機能** を使う
- インストール: VS Code の拡張機能タブで「Live Server」を検索してインストール
- 起動: `index.html` を右クリック →「Open with Live Server」
- URL: `http://127.0.0.1:5500/`
- メリット: ファイルを保存するたびにブラウザが自動リロードされる
- Pythonコマンド（`python3 -m http.server`）は使わない

**Node.js が必要になったとき（npmパッケージを使う場合など）**
→ Node.js を導入してから `npx` コマンドを使う
- Node.jsのインストールが必要になった時点で導入を検討する
- インストール: https://nodejs.org/ja/ からLTS版をダウンロード

**React（またはViteベースのフレームワーク）**
→ **Vite** でプロジェクトを作成する
```bash
npm create vite@latest プロジェクト名 -- --template react
cd プロジェクト名
npm install
npm run dev
```
- URL: `http://localhost:5173/`
- メリット: 高速なHMR（Hot Module Replacement）で変更が即反映される

### 各ステップ開始時
1. `learning-records/<課題名>/phaseN/step_N/` フォルダを作成
2. `learning-records/<課題名>/phaseN/step_N/lesson.md` を作成
3. `learning-records/<課題名>/phaseN/step_N/code/` フォルダを作成
4. 上記フォーマットに従って、詳細な説明を記録

### 各ステップで説明したとき
1. `phaseN/step_N/lesson.md` に以下を追記:
   - 考え方の説明（teaching.mdに基づく）
   - 段階的なヒント（hint.mdに基づく）
   - 公式ドキュメントのURL（reference-docs.mdに基づく）
   - コードの詳細な解説
   - よくあるエラーと対処法

### コード例を提示したとき
1. `phaseN/step_N/lesson.md` にコードを記録
2. `phaseN/step_N/code/<ファイル名>` として実際に動くコードファイルも保存

### エラーが出て対応したとき
1. `phaseN/step_N/lesson.md` の「よくあるエラー」セクションに追記
2. hint.mdに基づく3段階のヒントを記録

### ステップ完了時
1. `phaseN/step_N/lesson.md` の「学んだこと」セクションを記入
2. 参照した公式ドキュメントの一覧を記入
3. `progress.md` を更新（チェック・進捗カウント・今ここの移動）

---

## 記録の更新

- 説明や解説をするたびに、**その場で** `phaseN/step_N/lesson.md` に追記する
- 後でまとめて書くのではなく、リアルタイムで記録していく
- 受講生が後で読んだときに、学習の流れが分かるようにする
- **チャットで説明していた詳細な内容をすべてMDファイルに保存する**

---

## 完了時

- `progress.md` の全チェックボックスが埋まっている状態にする
- 各 `phaseN/step_N/lesson.md` が完成している状態にする
- 各 `phaseN/step_N/code/` フォルダにコード例が保存されている状態にする
